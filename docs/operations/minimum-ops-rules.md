# Skill Tree 最小运营规则

> 目标：把关键卡口钉死，避免团队后续因口径漂移、追数量、跳层接入而反复犯错。

---

## 1. 什么情况不能进 Shortlist

以下任一情况成立，**不得进入 shortlist**：

1. **技能身份不清晰**
   - 泛 AI 资源仓库
   - 普通教程/文章集合
   - 与 skill 复用场景关系弱

2. **基础元数据不完整**
   缺任一关键字段：
   - `repo`
   - `source`
   - `summary`
   - `category`
   - `safety_precheck`

3. **明显目录套目录且无新增价值**
   - 只是重复更大 registry / awesome list
   - 没有独立筛选价值或平台代表性

4. **已命中风险样本库的明确反例规则**
   - 同类高风险 / 低增量 / 低信号样本已被归档
   - 且当前条目没有新的战略价值说明

---

## 2. 什么情况不能从 Shortlist 进 Intake

以下任一情况成立，**不得进入 intake**：

1. **不属于 `优先收录`**
   - `观察` 和 `丢弃` 条目一律不得直升 intake

2. **风险等级为 `high`**
   - 默认不进正式收录候选
   - 除非裴谦明确拍板例外

3. **说不清两件事**
   - 为什么现在收
   - 收了以后对 Skill Tree 的直接价值是什么

4. **索引层无法消费**
   无法稳定补齐以下任一字段：
   - `platforms`
   - `priority_score`
   - `security_rating`
   - `include_reason`
   - `search_text`

---

## 3. 什么情况必须进风险样本库

以下任一情况成立，**必须进入风险样本库**：

1. **高风险默认阻断**
   - 代理外联
   - 部署/远程操作
   - 逆向工程
   - 多执行层委托
   - 安全研究高权限行为

2. **重复噪音但会反复出现**
   - 目录套目录
   - 弱 skill 身份项目
   - 低信号重复仓库

3. **本轮已被判定丢弃，且丢弃理由可复用**
   - 只要其失败原因能反向指导后续筛选，就必须入库

---

## 4. 谁负责卡口，超时怎么算

### Catalog → Shortlist
- **负责人**：林晚（研究）
- **卡口职责**：判断是否具备进入 shortlist 的最低条件
- **超时规则**：
  - 超过 1 个工作日未筛，默认继续留在 catalog
  - 不允许因赶进度跳过 shortlist 直接入 intake

### Shortlist → Intake
- **负责人**：林晚（研究）
- **卡口职责**：判断是否属于 `优先收录` 且具备正式收录理由
- **超时规则**：
  - 超过 1 个工作日未定，默认降级为 `观察`
  - 不允许无限悬挂在“快进 intake”状态

### Intake → Index
- **负责人**：辛海璐（总管）
- **卡口职责**：按 ingestion guide 补齐字段并写正式索引层
- **超时规则**：
  - 超过 1 个工作日未接入，退回 intake 待补字段检查
  - 不允许带缺字段强行入正式库

### 展示与产品层消费
- **负责人**：黄思博（产品）
- **卡口职责**：确保 UI 只消费正式索引层，不直接消费 shortlist / catalog
- **超时规则**：
  - 若字段未就绪，前端延后展示，不得绕过 index 直接读上游文件

---

## 5. 发现口径漂移时，按什么规则回滚到上一层

一旦出现以下任一情况，**必须回滚**：

1. **同类条目在 shortlist 与 intake 结论不一致**
   - 回滚到 shortlist 重新统一判断标准

2. **高风险条目被错误放入 intake 或正式库**
   - 立即回滚到风险样本库或观察池

3. **总管接入时发现字段定义不一致**
   - 回滚到 intake，按 ingestion guide 修正后再接入

4. **产品层发现正式索引与研究口径不一致**
   - 回滚到 index 上一层，不允许前端自行修补逻辑解释

### 回滚原则
- **字段问题** → 回滚到 intake
- **收录判断问题** → 回滚到 shortlist
- **风险分类问题** → 回滚到风险样本库归类规则
- **基础识别问题** → 回滚到 catalog

---

## 6. 一页执行原则

Skill Tree 后续执行统一遵守：

- 不跳层
- 不破例放 high risk 进正式库
- 不把观察项当正式收录
- 不让产品层直接消费非正式索引数据
- 一旦口径漂移，按“问题发生在哪一层，就回滚到哪一层”处理

---

## 结论

这套最小运营规则的作用只有一个：

**宁可慢一层，也不要把错的东西推进正式库。**

只要这条守住，Skill Tree 扩张时质量就不会塌。
