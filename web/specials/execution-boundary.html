30-second chooser

Start with Cloud sandbox if…

  • you want the fastest path to testing value
  • you can accept managed execution
  • your first question is “does this help?” rather than “can I fully govern it?”

Fit: fast evaluation, convenience, lower setup friction

Caution: easy does not mean transparent; verify where code and artifacts actually live.

30-second chooser

Start with Self-hosted if…

  • you need stronger control over runtime location
  • you care deeply about where secrets and data stay
  • you are willing to pay the ops cost for that control

Fit: customer-controlled infrastructure, stronger boundary clarity, durable runtime

Caution: self-hosted shifts the burden to you; it is not automatically safe without visibility and ops discipline.

30-second chooser

Start with Repo-native if…

  • you want the agent closest to your existing code workflow
  • you prefer explicit local/CI automation you can inspect
  • your team already works comfortably with scripts, hooks, and repo-local execution

Fit: local legibility, repo closeness, direct workflow attachment

Caution: repo-close automation can expand file-write and shell blast radius very quickly.

What usually goes wrong with Cloud sandbox

  • users do not realize how much is actually remote
  • secret and data assumptions are weaker than expected
  • execution evidence is thinner than needed for trust

What usually goes wrong with Self-hosted

  • teams underestimate setup and operational burden
  • self-hosted is treated as automatically safe without proper controls
  • observability is weaker than the deployment confidence suggests

What usually goes wrong with Repo-native

  • users over-trust local execution because it feels familiar
  • shell, file-write, and CI blast radius expands quickly
  • background durability is assumed but not actually supported
